## 1. Продуктовое видение

### Что такое GastroMap?

**GastroMap** — это AI-powered приложение для гастрономических открытий. Пользователь находит ресторан, кафе или бар не перебором списков, а через живой диалог с AI-ассистентом, который понимает контекст: "Хочу что-то уютное для свидания, без глютена, до 40 евро, рядом со Старым городом".

### Ключевая идея

> Не поисковик ресторанов, а **персональный гастрономический проводник**, который учится твоим вкусам, следит за здоровьем и помогает решить вечную проблему "куда пойти".

### Чем отличаемся от конкурентов

| Конкурент | Их подход | Наш ответ |
|---|---|---|
| Google Maps | Поиск + отзывы, нет персонализации | AI знает твои вкусы, не надо объяснять каждый раз |
| Yelp / TripAdvisor | Рейтинги и отзывы, UI перегружен | Простота: один вопрос → рекомендация |
| OpenTable | Только бронирование | Открытие + бронирование в одном месте |
| The Fork | Похожи, но нет AI | AI + контекст здоровья (Bio-Sync) — наш killer feature |

## 2. Позиционирование и бизнес-модель

### Целевые рынки

**Фокус MVP:** Краков (Польша) — плотная ресторанная сцена, туристический трафик, несколько языков.  
**Расширение V2:** Варшава, Вена, Прага, другие европейские города.

### Стратегия монетизации: два этапа

#### 🪙 Этап 1 — MVP: Донаты (сейчас)

На старте приложение полностью бесплатно. Монетизация через **добровольные донаты** — это честно, строит доверие и не создаёт барьер для первых пользователей.

**Механика донатов:**
- Кнопка "Поддержать проект" в профиле, в футере и на странице `/donate`
- Разовые суммы: €1 / €3 / €5 / своя сумма
- Провайдер: **Stripe** (разовый платёж, без подписки)
- После доната: красивый экран благодарности + бейдж "Supporter 💙" в профиле и лидерборде
- **Важно:** всё это через Stripe — значит вся платёжная инфраструктура уже настроена и готова к подпискам

**Почему это умно:**
- Stripe подключается один раз для донатов → переключение на подписки = только добавить продукт в Stripe Dashboard
- Бейдж Supporter создаёт социальное давление и виральность
- Прозрачность: страница `/about` показывает куда идут донаты (хостинг, AI API, развитие)

#### 💎 Этап 2 — V1.1+: Подписки (после MVP, архитектура уже готова)

Когда появится аудитория и станет ясно за что люди готовы платить — включается монетизация через подписки. **Вся архитектура уже заложена в коде как заглушка.**

**B2C Freemium:**
- **Free:** AI-чат (5 запросов/день), карта, фильтры, избранное
- **Premium — €4.99/мес / €39.99/год:** безлимитный AI, Bio-Sync, ранний доступ к фичам

**Топ-10 контрибьюторов** (по лидерборду) — вечная Premium бесплатно.

**B2B (рестораны) — V2:**
- **Basic Listing — бесплатно:** страница в каталоге
- **Boost — €49/мес:** повышение в AI-рекомендациях при релевантных запросах
- **Analytics — €99/мес:** статистика запросов, демография

**Транзакционная — V2:**
- Комиссия 5% с бронирований через приложение

> 🔑 **Ключевой принцип:** Stripe подключается один раз в MVP для донатов. Код подписок — заглушки с чёткими `// SUBSCRIPTION_GATE` комментариями. Включить подписки = раскомментировать + добавить Stripe Product ID.

## 5. Пользовательские сценарии (User Flows)

### Flow 1: Анонимный пользователь → Регистрация

```
[Landing Page]
  ↓ "Найти ресторан" / "Начать"
[Регистрация: Email или Google]
  ↓ Успешно
[Онбординг — Шаг 1/4: Кухни]
  → Выбирает 2-5 кухонь
[Онбординг — Шаг 2/4: Атмосфера]
  → Выбирает 1-3 виба (Romantic, Cozy, Trendy...)
[Онбординг — Шаг 3/4: Бюджет]
  → Выбирает уровень цен ($, $$, $$$)
[Онбординг — Шаг 4/4: Диета]
  → Опционально: вегетарианец, веган, без глютена...
[Dashboard — Главная]
  → AI сразу показывает персональные рекомендации
```

---

### Flow 2: Поиск через AI GastroGuide

```
[Dashboard / AI Guide страница]
  ↓ Пользователь вводит: "Хочу что-то уютное для свидания, недалеко от Рыночной площади"
[AI обрабатывает запрос]
  → detectIntent: 'recommendation'
  → search_locations(vibe=['Romantic','Cozy'], distance='near', price=['$$','$$$'])
  → Получает 3-5 мест
[Ответ AI с карточками локаций]
  ↓ Пользователь тапает на карточку
[Страница локации]
  ↓ "Сохранить" / "Был здесь" / "Забронировать"
[Подтверждение + очки начислены]
```

---

### Flow 3: Поиск через Карту

```
[Explore страница]
  ↓ Открывается карта с маркерами (текущий город)
[Пользователь тапает на фильтр]
  → Открывается FilterPanel (кухня, цена, диета, атмосфера, рейтинг)
  → Применяет фильтры → маркеры обновляются
[Тапает на маркер]
  → Появляется мини-карточка (снизу экрана)
  ↓ Тапает на мини-карточку
[Страница локации]
```

---

### Flow 4: Community Submission — предложить место

```
[Dashboard / любая страница]
  ↓ Кнопка "+" в Bottom Nav ИЛИ кнопка "Предложить место" в Profile
[Add Place Page — Шаг 1/3: Основное]
  → Название заведения (обязательно)
  → Адрес (с автодополнением через Nominatim)
  → Категория: Ресторан / Кафе / Бар / Пекарня
  → Город (автоопределяется из адреса)
  ↓ "Дальше"
[Add Place Page — Шаг 2/3: AI заполнение]
  → Кнопка "Заполнить с AI 🪄"
  → AI заполняет: описание, кухни, теги, атмосфера, диета, удобства
  → Пользователь проверяет и может редактировать
  → Must Try / Insider Tip — ТОЛЬКО вручную (AI не заполняет)
  ↓ "Дальше"
[Add Place Page — Шаг 3/3: Фото + подтверждение]
  → Загрузить 1-3 фото (опционально)
  → Превью карточки как она будет выглядеть
  → Чекбокс: "Подтверждаю, что это реальное место"
  ↓ "Отправить на проверку"
[Success экран]
  → "Спасибо! Проверим в течение 48 часов."
  → +100 очков после одобрения (в лидерборде)
  → Кнопка "Предложить ещё место"

[Admin — Moderation Queue]
  → Новое submission появляется с пометкой AI-check
  → AI уже проверил: нет дублей, адрес валидный, описание осмысленное
  → Admin: Approve (→ публикуется) / Reject (→ уведомление пользователю) / Edit&Approve
```

**Если функция выключена (feature flag `community_submissions = false`):**
```
  Кнопка "+" не отображается
  Маршрут /add-place → редирект на /dashboard
  (Без необходимости деплоя — только обновление feature_flags в БД)
```

### Flow 5: Групповой выбор / Anti-Solution (Заглушка)

```
[Страница локации или Dashboard]
  ↓ "Выбрать для группы"
[Групповая сессия]
```

```
│  AI-проверка: ✅ Нет дублей  ✅ Адрес валидный        │
│               ⚠️ Описание короткое (24 слова)          │
│                                                        │
│  [Просмотреть] [Edit & Approve ✅] [Отклонить ❌]     │
└────────────────────────────────────────────────────────┘
```

**При "Просмотреть"** → открывается Slide-over с полным превью (как выглядит карточка + все поля).

**При "Edit & Approve"** → открывается та же форма что у Admin для редактирования локации, с уже заполненными данными из submission. После сохранения → запись публикуется, пользователю +100 очков.

**При "Отклонить"** → модал с выбором причины:
- "Место уже есть в каталоге"
- "Недостаточно информации"
- "Адрес не найден / некорректен"
- "Нарушение правил"
- Своя причина (textarea)

После reject → пользователь получает уведомление с причиной.

### 7.13 Admin — Feature Flags (/admin/feature-flags)

Простая таблица переключателей без деплоя:

| Флаг | Описание | Состояние |
|---|---|---|
| `community_submissions` | Форма предложения мест для пользователей | ✅ ON / ❌ OFF |
| `donations` | Страница /donate и кнопки доната | ✅ ON |
| `subscription_plans` | Subscription Gate и тарифы | ❌ OFF (включить когда готово) |
| `bio_sync` | Подключение Apple Health / Google Fit | ❌ OFF |
| `voice_search` | Голосовой ввод в поиске | ❌ OFF |
| `dine_with_me` | Кнопка "Пригласить компанию" | ❌ OFF |
| `maintenance_mode` | Технические работы (блокирует app) | ❌ OFF |

**Как это работает в коде:**
- Таблица `feature_flags` в Supabase (id, key, enabled, description, updated_at)
- `useFeatureFlags()` хук — загружает флаги при старте, кэш 5 минут
- Все компоненты проверяют: `if (!flags.community_submissions) return null`
- Admin меняет toggle → мгновенно без деплоя

## 11. Архитектура AI-системы

### Компонент GastroGuide

```
[Пользователь] → Ввод запроса
     ↓
[Клиент] → POST /api/ai/chat { message, history, userContext }
     ↓
[Vercel Serverless] → buildSystemPrompt(userPrefs, foodieDNA)
     ↓
[OpenRouter API] → LLM с tool_use
     ↓
[LLM решает:] tool call "search_locations"?
     ├── YES → возвращает tool call JSON
     │          ↓
     │   [Клиент] executeTool → фильтрует locations Zustand store
     │          ↓
     │   [Клиент] → отправляет результаты обратно в /api/ai/chat
     │          ↓
     │   [LLM] → финальный ответ с рекомендациями
     └── NO → сразу финальный ответ
     ↓
[Клиент] → рендер AI-сообщения + Location Cards
```

## 15. Community Submissions — полная архитектура

### 15.1 Концепция и цели

**Зачем это нужно на MVP:**
- Наполнение базы данных силами community — снижает нагрузку на админа
- Engagement: пользователи чувствуют причастность к росту продукта
- Виральность: "Я добавил это место!" → рассказывает друзьям
- Лидерборд: +100 очков за одобренное место → мотивация

**Управляется feature flag `community_submissions`:**
- `true` — кнопка "+" в Bottom Nav, маршрут `/add-place` работает
- `false` — кнопка скрыта, маршрут редиректит на Dashboard, **без деплоя**
- Идеально для: временного отключения во время технических работ, A/B тестов, регионального ограничения

### 15.3 AI-препроверка при submit

Сразу после отправки формы — до попадания в очередь модератора — AI автоматически проверяет submission:

```js
// api/submissions/check.js (Vercel Serverless)

async function checkSubmission(submission) {
  return {
    // 1. Проверка дублей (поиск похожих названий в БД)
    duplicates: await findSimilarLocations(submission.name, submission.city),
    // → если найдено: предупреждение модератору

    // 2. Валидация адреса через Nominatim
    address_valid: await validateAddress(submission.address, submission.city),
    // → если false: флаг для модератора

    // 3. Оценка качества контента (0-100)
    quality_score: await scoreContent(submission),
    // description длиннее 50 слов = +20 очков
    // must_try заполнено = +25 очков
    // insider_tip заполнено = +25 очков
    // фото приложены = +30 очков

    // 4. Категоризация риска
    risk_level: 'low' | 'medium' | 'high'
    // high = дубли найдены ИЛИ адрес невалиден → требует внимания
  }
}
```

Результат сохраняется в `user_submissions.ai_check_result` и показывается в AdminSubmissionsPage.

### 15.4 RLS политики для Submissions

```sql
-- Пользователь видит только свои submissions
CREATE POLICY "users_see_own_submissions"
ON user_submissions FOR SELECT
USING (auth.uid() = user_id);

-- Пользователь может создавать submissions (если feature flag включён)
CREATE POLICY "users_can_submit"
ON user_submissions FOR INSERT
WITH CHECK (auth.uid() = user_id);

-- Admin видит все, может обновлять статус
CREATE POLICY "admin_full_access_submissions"
ON user_submissions FOR ALL
USING (get_my_role() = 'admin');
```

### 15.5 Trigger для очков при approve

```sql
-- Функция: начислить очки при одобрении submission
CREATE OR REPLACE FUNCTION handle_submission_approved()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.status = 'approved' AND OLD.status != 'approved' THEN
    -- Начисляем 100 очков через запись в user_visits или отдельную таблицу
    INSERT INTO user_contribution_points (user_id, points, reason, reference_id)
    VALUES (NEW.user_id, 100, 'submission_approved', NEW.id)
    ON CONFLICT (user_id, reference_id) DO NOTHING;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_submission_approved
  AFTER UPDATE ON user_submissions
  FOR EACH ROW EXECUTE FUNCTION handle_submission_approved();
```

### 16.5 SubscriptionGate компонент (код написан, gate выключен)

```jsx
// src/shared/components/guards/SubscriptionGate.jsx

/**
 * SUBSCRIPTION_GATE
 * Пока подписки не активированы (feature_flags.subscription_plans = false),
 * этот компонент просто рендерит children без ограничений.
 * 
 * Активировать подписки:
 * 1. Создать Stripe Products и скопировать Price IDs
 * 2. Добавить price IDs в subscription_plans таблицу
 * 3. Установить feature_flags.subscription_plans = true
 * 4. Установить feature_flags для конкретной фичи (bio_sync, etc.)
 * 5. Убрать early return ниже
 */

function SubscriptionGate({ feature, children, fallback }) {
  const { flags } = useFeatureFlags();
  const { user } = useAuthStore();

  // SUBSCRIPTION_GATE: убрать эту строку для активации
  if (!flags.subscription_plans) return children;

  const hasAccess = checkUserAccess(user, feature);
  if (!hasAccess) return fallback || <UpgradePrompt feature={feature} />;
  return children;
}

// Использование в коде (уже вставлено везде где будут ограничения):
// <SubscriptionGate feature="ai_unlimited">
//   <AIChat unlimited />
// </SubscriptionGate>
```
